## Why This Exists

Every AI tool puts you in the orchestrator seat тАФ you decide which model, what context to paste, what to carry across sessions, when to switch tools. For a system that's supposed to be working _for_ you, that's a lot of operator work.

JARVIS flips it. You say what you want in plain language. An orchestrator routes the request to the right specialist. The specialists have their own tools, their own memory, and their own rules. The system extracts a lesson from every failure and never makes the same mistake twice.

It's not an assistant. It's a team that happens to live inside Claude Code.

JARVIS never asks the same question twice. Memory is stored in structured, capped markdown files тАФ loaded lazily so they don't bloat every conversation.

## ЁЯТм Examples

<details>
<summary><b>ЁЯФм Deep Research</b></summary>

Ask for a deep dive and you get a 6-phase pipeline, not a one-shot answer:

1. **Scope** тАФ JARVIS restates the question and flags ambiguities
2. **Search** тАФ 6-10 parallel queries across sources
3. **Screen** тАФ filter to high-signal sources, discard junk
4. **Extract** тАФ pull claims, stats, quotes with citations
5. **Synthesize** тАФ build the narrative, flag gaps
6. **Deliver** тАФ 1,500тАУ3,000 word report with source list + confidence levels

Claims that can't be verified get flagged with `[unverified]` instead of being silently presented as fact.
</details>

<details>
<summary><b>ЁЯФи Building an App</b></summary>

The `vibecode-app-builder` skill runs a 25-prompt process over 7 working days:

- **Day 1** тАФ PRD, stack decision, schema
- **Day 2** тАФ File structure + auth
- **Day 3** тАФ Core feature + self-healing integrations
- **Day 4** тАФ Dashboard + UI polish
- **Day 5** тАФ Payments (Stripe)
- **Day 6** тАФ Mobile-friendly polish
- **Day 7** тАФ Deploy + monitoring

Every step runs through the self-healing executor: build тЖТ test тЖТ diagnose тЖТ repair тЖТ retry, up to 5 iterations before escalating to you.
</details>

<details>
<summary><b>ЁЯзм Self-Improvement in Action</b></summary>

A real pattern from usage:

> JARVIS tries Firecrawl on a G2 review page тЖТ rate-limited тЖТ falls back to Chrome scrape тЖТ succeeds.
>
> MetaClaw extracts the lesson:
> - **Rule:** G2/Capterra review pages rate-limit Firecrawl тАФ use Chrome scrape instead
> - **Confidence:** HIGH
> - **Stored in:** `skills/learned/tool-routing.md`
>
> Next time anyone asks JARVIS to scrape G2, it goes straight to Chrome. The failure never repeats.
</details>

## ЁЯТб Philosophy

> *Five principles that drive every design decision.*

**1 тАФ Route, don't execute.** The orchestrator never does domain work directly. It delegates to specialists. Context stays clean, outputs stay high-quality.

**2 тАФ Memory is sacred.** The system is only as good as what it remembers. Memory files are capped, structured, and maintained carefully тАФ so information stays current and never bloats.

**3 тАФ Self-improve or stagnate.** JARVIS extracts lessons from every failure. The MetaClaw system turns mistakes into permanent rules. It's not just an assistant тАФ it's a learning system.

**4 тАФ Plan before executing.** For any task over 3 steps, JARVIS proposes a plan first. For significant work, it writes a Requirement Document. Wasted execution is worse than wasted planning.

**5 тАФ The most direct tool wins.** Direct MCP over browser automation. Specific API over general search. Fastest reliable path тАФ always.
